## 1. 项目定位

**GameToolX** — 一个多语言游戏工具聚合站，单机部署或云自部署，**主目标**：

- 东南亚（vi/th/id，phase 2）+ 日韩 + 台湾 + 简体中文
- 工具驱动：每个游戏至少 2 个工具（walkthrough + system-checker 起，扩展按游戏特点加）
- 不做社交，不做账号系统，不做云存档
- 内容来源真实可溯（no fabrication，2+ source cross-validate）

**目标规模**：100 并发 / 10k DAU（SQLite WAL 撑得住）

## 3. 数据源规约（强约束）

### 主源

- **Pser_hanser 攻略文档**（飞书 doc，199KB / 3002 行）— SMT5V 攻略、boss、机制
- **Fandom MediaWiki API**（如 megaten.fandom.com）— 恶魔/技能数据
- **Steam Store API** — 元数据（价格、tag、review）
- **官方 ATLUS/SEGA 页面** — 媒体素材

### 源数据规则

1. **不能编造**：所有技能/伤害数字/概率必须能从 2+ 源交叉验证
2. **顶层 sources**：每条 tool 数据根上挂 `sources: [{lang, url?, attribution}]`
3. **块级 sources**：每个内容块可挂 `sources?: [{lang, url, attribution}]`
4. **url 可选**（lib/data.ts v2026-06-08）：源 doc 临时 blob URL 不必存
5. **翻译内容**：`crawled_chunks.translations` 是 JSON map，加性存储；查 `searchChunks({preferredLang})` 优先返回翻译后文

### 不能做的事

- **不写 AI 生成的图片**进前端（图片只能从 Wiki / Steam / 官方 / 玩家 doc）
- **不写 doc 截图**进 walkthrough 块（"图片不要截图"，用结构化文本）
- **不暴露 LLM 标注痕迹**："tagged by LLM" / "AI 标注" 字样不进前端

## 7. 存储架构（v2.1，2026-06-09 升级）

### 图像存储分两轨

**轨道 1 — git tracked**（11 个文件，~800KB）：
```
public/images/games/{slug}/
├── library.jpg      # 封面
├── header.jpg
├── capsule.jpg
├── logo.png
├── hero.jpg
└── screenshots/ss_00.jpg ~ ss_05.jpg
```

**轨道 2 — COS-backed via symlink**（332 文件，~120MB）：
```
/www/wwwroot/gametoolx.top/public/images/games/smt5v/
├── [track-1 文件]                    ← 上面 11 个
├── bosses/  → /cos-images/bosses     ← 软链
├── regions/ → /cos-images/regions
├── chests/  → /cos-images/chests
├── demons/  → /cos-images/demons
└── misc/    → /cos-images/misc
```

### COS 配置

- **桶**: `gametoolx-1440949724`（ap-tokyo，标准存储，10GB 免费额度内）
- **挂载**: `/cos-images`（256TB cosfs FUSE 容量）
- **挂载进程**: `cosfs gametoolx-1440949724:/ /cos-images -ourl=http://cos.ap-tokyo.myqcloud.com -odbglevel=err -oallow_other -opublic_bucket=1 -oensure_diskfree=10240`
- **桶内组织**: `bosses/`、`regions/`、`chests/`、`demons/`、`misc/`（5 个 flat 目录）

### 同步脚本

- `scripts/sync-images.sh {push|pull|status}`
- 默认配置：本地 → 服务器（rsync over SSH）
- push 时自动 `mkdir -p` 5 个子目录（恢复 `git pull` 删子目录的边角情况）

### git pack 减肥效果

| 指标 | 之前 | 之后 |
|---|---|---|
| git pack | 29MB | 几 KB（新 commit）|
| 本地图片占用 | 120MB | 1.2MB |
| git push 耗时 | 5+ 分钟（GCM 慢）| < 1 秒 |

### ⚠️ 已知 gotcha（已写进 sync-images.sh 头部）

`git pull` 在新 commit 解除跟踪时，**会删除工作区对应文件**（git 默认行为，不是 bug）。
**所以**：
- 服务器 `git pull` 任何解除跟踪的 commit → 5 子目录被删
- 恢复：`bash scripts/sync-images.sh push`（本地有图的前提下）
- 长期缓解：用 git LFS 或迁独立 store

## 16. SystemChecker 重构计划（2026-06-09 新增需求）

### 目标

1. **进入页面自动检测**：移除"开始检测"按钮，页面加载后自动检测并在结果区显示结论
2. **多游戏支持 + 默认填充**：从不同游戏进入系统检测时，自动选中当前游戏；用户可在 UI 中切换其他游戏

### 当前架构（重构前）

- 每个 game 独立 tool slug：`{game-slug}-system-checker`（e.g. `octopath-traveler-2-system-checker`）
- dispatcher 按 slug → tool → tool.gameSlug → game
- SystemCheckerClient 有 `autoTrigger` prop（默认 false）+ `?game=` URL 切换 + game picker，**但** dispatcher 没传 `autoTrigger`，所以**没自动检测**
- 已有 8 成代码，只是没串联

### 重构后架构

```
通用工具入口:
  /[lang]/tools/system-checker             # 默认 game（games.json[0]）
  /[lang]/tools/system-checker?game=xxx   # 预选 game

game page tool card:
  /[lang]/games/[slug]                     # 工具卡片链接改成
  <Link href={`/${lang}/tools/system-checker?game=${game.slug}`}>

dispatcher:
  slug === "system-checker" (通用):
    - 读 searchParams.game（默认 games[0].slug）
    - getGame(gameSlug) 拿 game 数据
    - autoTrigger={true} 传进 SystemChecker
    - JSON-LD 标题不带 "游戏名 -" 前缀（或保留通用标题）
  其他 slug:
    - 走原 per-game 逻辑（不影响）
```

### 改动清单

1. **`data/tools/system-checker.json`** — 创建通用工具元数据
   ```json
   {
     "slug": "system-checker",
     "gameSlug": null,  // 通用，不绑 game
     "type": "system-checker",
     "title": { "en": "System Requirement Checker", "ja": "動作環境チェッカー", "ko": "시스템 요구사항 체커", "zh": "系统配置检测" },
     "description": { ... }
   }
   ```

2. **`app/[lang]/tools/[slug]/page.tsx`** — 改 dispatcher
   - 加 `searchParams` prop
   - slug === "system-checker"：拿 query game（默认 first game），传 `autoTrigger={true}`，title 通用
   - 保留 per-game slug 逻辑（向后兼容，可能有外链）

3. **`app/[lang]/games/[slug]/page.tsx`** — 改 tool link
   - system-checker 类型 → link 改为 `/tools/system-checker?game={gameSlug}`
   - 其他类型不动

4. **`data/games/*.json`** — 更新 `availableTools` 和 `tools`
   - `availableTools: [..., "system-checker"]`
   - `tools: [..., "system-checker"]`
   - 删除 game-specific 旧条目（`octopath-traveler-2-system-checker`）
   - 或者保留旧条目（向后兼容），但 game page 不再链

5. **`components/tools/SystemCheckerClient.tsx`** — 改自动检测行为
   - `useEffect` 默认触发 `handleAutoDetect`
   - 触发后**自动** `setSubmitted(true)`（让结果区立即显示）
   - 若 GPU 未识别：结果区显示 "请输入 GPU 型号" 提示，form 区仍可填
   - 把 "🔍 Auto-detect" 按钮文案改成 "🔄 重新检测"
   - form 提交按钮文案改成 "重新计算"

6. **CSS / 视觉**：
   - 页面加载时给结果区加个 "正在检测..." 动画（200-500ms 模拟）
   - 检测结果到位时滚动到结果区

### 数据流（重构后）

```
[Game Page] 点 "系统检测" 卡片
  ↓
/zh/tools/system-checker?game=shin-megami-tensei-5-vengeance
  ↓
dispatcher:
  - slug=system-checker → 通用分支
  - searchParams.game=shin-megami-tensei-5-vengeance
  - getGame(gameSlug) → game data
  - autoTrigger={true}
  ↓
SystemCheckerClient (mounted):
  - useEffect → handleAutoDetect()
  - detectHardware() 读 navigator.hardwareConcurrency / deviceMemory / platform / screen / UA
  - 填表 (CPU 推测 + RAM 推测)
  - setSubmitted(true) 立即计算（GPU 未知给 partial）
  - 显示结果区
  ↓
用户切换 game picker:
  switchGame(other-slug) → router.push(...?game=other-slug)
  → 重新进入流程，game 数据换
```

### 测试场景

1. **首页直接进**：`/zh/tools/system-checker` → 默认第一个 game，autoTrigger
2. **从 OT2 进**：`/zh/tools/system-checker?game=octopath-traveler-2` → OT2，autoTrigger
3. **从 SMT5V 进**：`/zh/tools/system-checker?game=shin-megami-tensei-5-vengeance` → SMT5V，autoTrigger
4. **切换游戏**：UI 选 OCT → URL 变 → game 数据换 → 重检测
5. **手动填 GPU**：填完 RTX 4060 → 点"重新计算" → 结果更新
6. **UA 推断 GPU**：如果 userAgent 含 "RTX 4070"，auto-detect 时 GPU 也填上 → 完美结果

### 风险

- **静态生成 (SSG)**：`generateStaticParams` 只对固定 slug 预渲染，通用 system-checker 是动态？→ 没事，Next.js 会 SSG 默认 game，其他 game 走 ISR
- **URL 污染**：搜索引擎可能把 `?game=xxx` 当独立页面索引 → 加 canonical 指向 `/tools/system-checker`
- **多语言文案**：4 语言需要把新文案（"正在检测..." / "重新计算"）都加到 `data/i18n/*.json`
